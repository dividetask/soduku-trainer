# Tech Stack

This document records the implementation stack for the Android app. It
captures the decision and the reasoning so later contributors don't have
to re-derive it.

## Decision

**Kotlin + Jetpack Compose**, building a native Android app.

| Concern        | Choice                                                     |
|----------------|------------------------------------------------------------|
| Language       | Kotlin (latest stable)                                     |
| JDK            | 17                                                         |
| UI toolkit     | Jetpack Compose + Material 3                               |
| Navigation     | Navigation-Compose (two destinations: `MainMenu`, `Game`)  |
| State          | `ViewModel` + `StateFlow<GameState>`; immutable data classes |
| Min SDK        | API 26 (Android 8.0)                                       |
| Target SDK     | Latest stable at time of release                           |
| Build          | Gradle with Kotlin DSL                                     |
| Unit tests     | JUnit + `kotlin.test` on the pure-logic layer              |
| UI tests       | Compose UI test framework (`createAndroidComposeRule`)     |

## Why Kotlin

- Google's official first-class Android language; all new Android APIs
  and documentation lead with Kotlin.
- Null safety, data classes, and `sealed interface` map almost 1:1 onto
  the shapes in [`data-model.md`](data-model.md). For example:
  ```kotlin
  sealed interface Move
  data class SetValueMove(...) : Move
  data class TogglePencilMarkMove(...) : Move
  ```
- Far less boilerplate than Java for the same behavior, which matters on
  a project this small.

## Why Jetpack Compose

- The Game Screen is naturally a pure function of `GameState`: Compose
  re-renders only the parts that changed when state updates, which
  removes a whole category of "view got out of sync with model" bugs.
- Custom drawing — the 9x9 board, pencil marks in fixed 3x3 slots,
  highlighted digits — is straightforward with `Canvas` or a grid of
  `Box` composables. No `findViewById`, no XML-plus-view-code split.
- `remember`, `mutableStateOf`, and `collectAsState` compose cleanly
  with a `ViewModel` holding a `StateFlow<GameState>`.
- Previews and live edit make iterating on board visuals fast.

## Alternatives Considered

- **Java + XML views** — Rejected. More boilerplate, older idioms, and
  the custom board UI is more painful in imperative XML.
- **Flutter / React Native** — Rejected. Cross-platform cost is wasted
  on an Android-only product and adds a platform layer we don't need.
- **Kotlin Multiplatform + Compose Multiplatform** — Rejected for v1.
  Viable if iOS/desktop ever become targets, but the extra setup buys
  nothing today. Worth revisiting if platform scope grows.
- **A game engine (Unity, Godot)** — Rejected. This is a UI-bound app,
  not a real-time renderer; a game engine is a mismatch.

## Module / Package Layout (proposed)

A single Gradle module is enough for v1. Suggested package structure:

```
com.dividetask.sudokutrainer
├── app               // Application, DI wiring
├── ui
│   ├── menu          // MainMenuScreen, MainMenuViewModel
│   ├── game          // GameScreen, GameViewModel, board composables
│   └── theme         // Material 3 theme, color tokens
├── domain
│   ├── model         // GameState, Cell, PencilMark, Move (sealed)
│   ├── puzzle        // Puzzle generator / loader
│   └── solver        // (v2) technique-based solver for hints
└── util              // Small pure helpers
```

Keeping `domain` free of Compose / Android dependencies lets us unit-
test the generator, move application, and (eventually) the solver
without spinning up an emulator.

## State Management Sketch

```kotlin
class GameViewModel : ViewModel() {
    private val _state = MutableStateFlow(GameState.forNewPuzzle(...))
    val state: StateFlow<GameState> = _state.asStateFlow()

    fun onDigitSelected(digit: Int) { ... }
    fun onCellTapped(row: Int, col: Int) { ... }
    fun onPencilToggled() { ... }
    fun onColorSelected(color: Color) { ... }
    fun onUndo() { ... }
    fun onGiveUpOrSolve() { ... }
}
```

Every handler either mutates non-board UI state (digit, color, pencil)
or produces a `Move`, applies it, and pushes it onto the history — the
same history Undo pops from. See [`game-mechanics.md`](game-mechanics.md)
and [`data-model.md`](data-model.md).

## v1 Puzzle Source

Puzzles in v1 are **pre-generated by us** (no third-party datasets, no
licensing obligations) and bundled with the app as a static asset.

- **Pool size**: **150** puzzles for v1.
- **Asset location**: `app/src/main/assets/puzzles/v1.json`, parsed on
  demand into the `Puzzle` shape from [`data-model.md`](data-model.md).
- **How they're generated**: a small **offline generator tool** — a
  standalone Kotlin program living under `tools/puzzle-gen/` (or run as
  a `main`-having Gradle task) — produces the JSON once. It:
  1. Generates a random complete 9x9 grid via randomized backtracking.
  2. Removes clues one at a time, re-running a uniqueness-checking
     solver after each removal and restoring the clue if the puzzle
     would become ambiguous.
  3. Repeats until the target clue count is reached, then serializes
     `id`, `difficulty`, `givens`, and `solution` to JSON.
- **When it runs**: manually, by a developer, whenever we want to
  refresh or expand the pool. The generator is **not** shipped inside
  the app APK; only its output is.
- **Puzzle selection at runtime**: picking a puzzle from the pool is a
  simple random choice from the 150-entry list. Avoiding immediate
  repeats is a nice-to-have but not required for v1.

v3 will add an on-device generator; the `Puzzle` shape is identical so
the rest of the app does not have to change when the source switches.

## Open Stack Decisions

Not blocking v1; call-outs for later:

- **Dependency injection**: Hilt is the default for Android Compose
  apps. For a single-module app this size we can also just use manual
  construction; no strong reason to pick yet.
- **Persistence**: None in v1. If we add save/resume, Jetpack DataStore
  (Proto) is the natural fit.
